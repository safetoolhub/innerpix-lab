### 2. Objetivo del Refactor

**Problema Principal:**
El archivo `main_window.py` actual viola el principio de responsabilidad única (SRP), conteniendo lógica de negocio, gestión de estado, manejo de eventos UI y coordinación de servicios en un único módulo acoplado.

**Objetivo:**
Realizar un refactor arquitectónico completo aplicando principios SOLID para desacoplar `main_window.py`, reducir su complejidad y mejorar la mantenibilidad del código sin alterar la funcionalidad, diseño visual ni experiencia de usuario

### 3. Restricciones Críticas del Refactor

**Invariantes (NO DEBEN cambiar):**

- ✅ Funcionalidad exacta para el usuario final
- ✅ Interfaz gráfica y disposición de elementos
- ✅ Flujo de interacción y usabilidad
- ✅ Comportamiento de todas las características existentes
- ✅ Formato de logs y archivos de salida

**Permitido:**

- ✅ Creación de nuevas clases y módulos

- ✅ Mejora de la arquitectura sin impacto en la UI


### 4. Principios Arquitectónicos a Aplicar

#### 4.1 Separación de Responsabilidades (SRP)

- **MainWindow:** Solo gestión de la interfaz gráfica y delegación de eventos
- **Controllers:** Coordinación entre UI y servicios de negocio
- **Services:** Lógica de negocio pura (ya implementados en `/services/`)
- **Models:** Gestión de estado y datos de la aplicación
- **ViewModels/Presenters:** Preparación de datos para la vista


#### 4.2 Principio de Inversión de Dependencias (DIP)

- MainWindow debe depender de abstracciones, no de implementaciones concretas
- Implementar interfaces/protocolos cuando sea necesario para desacoplar componentes


#### 4.3 Patrón Command para Operaciones

- Encapsular cada operación (renombrar, detectar duplicados, limpiar) como comandos ejecutables[^5]
- Facilitar implementación futura de undo/redo

### 6. Estrategia de Refactorización por Fases

#### Fase 1: Extracción de Modelos

- Crear clases de modelo para representar el estado de la aplicación
- Extraer toda la lógica de gestión de estado de `main_window.py`


#### Fase 2: Creación de Controladores

- Implementar controladores que coordinen entre UI y servicios
- MainWindow delegará toda la lógica de negocio a estos controladores


#### Fase 3: Refactorización de MainWindow

- Reducir `main_window.py` a:
    - Inicialización de la UI
    - Conexión de señales a métodos de controladores
    - Actualización de la UI basada en modelos
- Métodos de MainWindow deben ser principalmente one-liners que deleguen


#### Fase 4: Implementación de Señales Centralizadas

- Crear un sistema de señales personalizado para comunicación entre componentes
- Evitar acoplamiento directo entre vistas y controladores


### 7. Patrones de Diseño Recomendados[^5]

- **MVC/MVP:** Separación clara entre Vista, Modelo y Controlador/Presenter
- **Command:** Para operaciones reversibles y auditables
- **Observer:** Para notificaciones de cambios de estado (PyQt signals/slots)
- **Factory:** Para creación de workers y servicios si es necesario
- **Strategy:** Para diferentes algoritmos de detección de duplicados


### 8. Criterios de Calidad del Refactor[^7][^8]

**Métricas de Éxito:**
